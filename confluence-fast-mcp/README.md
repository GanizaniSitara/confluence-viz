# Confluence Fast MCP Server

FastMCP server for serving Confluence data from pickled files. 10-100x faster than live API calls.

Supports 30+ concurrent users for hackathons and team collaboration.

**For hackathons**: See [HACKATHON.md](HACKATHON.md) for quick setup guide.

## Prerequisites

- Pickled Confluence data in `../temp/` (generated by `sample_and_pickle_spaces.py`)
- **Python servers**: Python 3.8+, FastMCP 3.0+
- **Go servers**: Go 1.21+ (no Python needed at runtime)

## Installation

### Python

```bash
cd confluence-fast-mcp
pip install fastmcp beautifulsoup4 lxml pydantic requests python-dateutil
# Optional, for full-text search server:
pip install whoosh3
```

### Go

```bash
# Simple server (in-memory search)
cd confluence-fast-mcp/simple-server-go
go build -o simple-server .

# Full-text server (TF-IDF index, replaces WHOOSH)
cd confluence-fast-mcp/server-go
go build -o server .
```

Both read `.pkl` files directly from `../temp/` and default to HTTP on port 8070 — same as the Python versions.

## Configuration

Copy the example settings file and edit if needed:

```bash
cp settings.example.ini settings.ini
```

Default configuration (works if pickles are in `../temp/`):
```ini
[data]
pickle_dir = ../temp
index_dir = ./whoosh_index
```

## Usage

### Option 1: Simple Server (Recommended for Getting Started)

Just loads pickles into memory - instant startup, no indexing required.

**Start server:**
```bash
python simple_server.py
```

Listens on `http://0.0.0.0:8070` by default. Use `--port 9000` to change, or `--stdio` for Claude Desktop.

Features:
- Instant startup (loads 80K pages in seconds)
- In-memory title search
- All basic MCP tools work
- No WHOOSH dependency
- Supports 30+ concurrent users

### Option 2: Full Server with WHOOSH Search

Build search index first, then run:

```bash
python build_index.py
python server.py
```

Features:
- Full-text search with CQL
- Index build required (can take time for large datasets)
- Advanced search capabilities

### Option 3: Go Simple Server (fastest startup)

```bash
cd simple-server-go && go build -o simple-server .
./simple-server                           # reads pickles from ../temp
```

Reads `.pkl` files directly (via gopickle). Same 5 MCP tools as the Python version, compiled to a single binary.

### Option 4: Go Full-Text Server (TF-IDF index, no WHOOSH)

```bash
cd server-go && go build -o server .
./server                                  # reads pickles, builds index on first run
```

Or build the index separately:
```bash
./server build-index --data-dir ../temp --index-dir ./index_data
./server --data-dir ../temp --index-dir ./index_data
```

Features:
- Reads `.pkl` files directly (no conversion step)
- TF-IDF ranked full-text search (replaces WHOOSH)
- Index builds in ~2s (vs 10-30s for WHOOSH)
- Single binary, zero Python dependencies
- Same tool output format as Python servers

### Add to Claude Desktop

Add to your MCP settings file:

**macOS**: `~/Library/Application Support/Claude/claude_desktop_config.json`
**Windows**: `%APPDATA%\Claude\claude_desktop_config.json`
**Linux**: `~/.config/Claude/claude_desktop_config.json`

**Simple server** (no indexing):
```json
{
  "mcpServers": {
    "confluence-simple": {
      "command": "/usr/bin/python",
      "args": ["/home/user/git/confluence-viz/confluence-fast-mcp/simple_server.py", "--stdio"]
    }
  }
}
```

**Full server** (with WHOOSH):
```json
{
  "mcpServers": {

      "command": "/usr/bin/python",
      "args": ["/home/user/git/confluence-viz/confluence-fast-mcp/server.py", "--stdio"]

  }

```

**Go simple server**:
```json

    "confluence-simple": {
      "command": "/home/user/git/confluence-viz/confluence-fast-mcp/simple-server-go/simple-server",
```

Update paths to match your actual installation.

### Add to Claude Code (CLI)

**Step 1: Start the server** (runs independently, visible via ps):

```bash
python simple_server.py
```

**Step 2: Configure Claude Code** - Edit `~/.claude/mcp_settings.json`:

```json
{
  "mcpServers": {
    "confluence-simple": {
      "transport": "http",
      "url": "http://localhost:8070/sse"
    }
  }
}
```

The server keeps running independently. Multiple Claude Code instances can connect to it.

### Restart

Restart Claude Desktop or Claude Code for changes to take effect.

## Available MCP Tools

- `getConfluenceSpaces()` - List all spaces
- `getConfluencePage(pageId)` - Get page with ADF body
- `getPagesInConfluenceSpace(spaceKey)` - List pages in space
- `searchConfluenceUsingCql(cql)` - Search with CQL syntax
- `search(query)` - Simple search
- `fetch(ari)` - Fetch by Atlassian Resource Identifier

## Troubleshooting

**"No pickle files found"**
- Run `sample_and_pickle_spaces.py` from parent directory first
- Check `pickle_dir` in `settings.ini` points to correct location

**"Module not found"**
- Install dependencies: `pip install fastmcp beautifulsoup4 lxml pydantic requests`
- Make sure you're running from the `confluence-fast-mcp/` directory

**"Index not found" / "Index is empty"**
- Run `python build_index.py` to create the search index
- This must be done before starting the server

## Performance

|                 | Python (WHOOSH) | Go (TF-IDF) |
|-----------------|-----------------|--------------|
| Index build     | 10-30s          | ~2s          |
| Search          | <100ms          | <10ms        |
| Page retrieval  | <10ms           | <1ms         |
| Startup (warm)  | ~5s             | <1s          |

Both are 10-100x faster than live Confluence API calls.

## Windows Performance Guidance (Before a Go Rewrite)

If your primary goal is **faster response time**, use this decision order:

1. **`simple_server.py` first** (no index build, fastest startup, lower overhead)
2. **`server.py` + WHOOSH** only when you need CQL/full-text queries

WHOOSH is good for an embedded Python index, but it is **not the highest-performance option** on Windows for large full-text workloads. Before rewriting in Go, also consider:

- **SQLite FTS5** (very fast embedded search, simple deployment)
- **Meilisearch/OpenSearch/Elasticsearch** (best query performance at scale, but extra service to run)
- **Running current Python server in WSL2** (often faster filesystem/index performance than native Windows Python)

If we do move to Go, keep both variants to match current behavior:

- `go/simple-server/` (equivalent to `simple_server.py`)
- `go/full-server/` (equivalent to `server.py` with indexed full-text search)

## Architecture

### Python servers (project root)

| File | Purpose |
|------|---------|
| `simple_server.py` | FastMCP server, in-memory search (recommended) |
| `server.py` | FastMCP server with WHOOSH full-text search |
| `build_index.py` | Builds WHOOSH index from pickles |
| `convert_to_json.py` | Optional: converts pickles to JSON |
| `config.py` | Configuration management (`settings.ini`) |
| `pickle_loader.py` | Loads and indexes pickled Confluence data |
| `converters.py` | HTML to Markdown/ADF conversion |
| `search.py` | CQL query parsing |
| `indexer.py` | WHOOSH full-text search index |
| `fallback.py` | Optional live Confluence API client |
| `models.py` | Pydantic response models |

### Go servers

| Directory | Purpose |
|-----------|---------|
| `simple-server-go/` | Go replica of `simple_server.py` (in-memory search) |
| `server-go/` | Go replica of `server.py` (TF-IDF inverted index, replaces WHOOSH) |

Both Go servers read `.pkl` pickle files directly (via gopickle) — no conversion step needed.
Same 5 MCP tools, same output format — drop-in replacements for the Python versions.
